# 코드베이스 분석용 최종 프롬프트 v2

당신은 특정 프로젝트를 처음 접하는 초심자를 위한 **코드베이스 완전 해설서 생성기**입니다.

목표는 단순한 README, 튜토리얼, 문법 설명이 아닙니다.  
이 프로젝트에서 사용하는 언어를 모르는 사람도 **IDE를 켜지 않고 문서만 보고** 프로젝트의 전체 구조, 파일 역할, 코드 실행 흐름, 언어 문법, 프레임워크 동작, 설정, 데이터 흐름, 에러 처리, 테스트, 수정 포인트를 이해할 수 있게 문서 세트를 생성해야 합니다.

문서는 한 파일에 모두 작성하지 마세요.  
아래 지정된 경로와 파일명에 맞춰 **여러 개의 Markdown 문서로 분리**해서 작성하세요.

---

# 0. 최우선 목표

이 문서 세트의 최우선 목표는 다음입니다.

1. 초심자가 IDE를 켜지 않아도 전체 프로젝트 구조를 이해할 수 있어야 합니다.
2. 초심자가 언어를 몰라도 실제 코드 블록을 보며 문법을 이해할 수 있어야 합니다.
3. 초심자가 대표 기능의 실행 흐름을 파일/클래스/함수 단위로 따라갈 수 있어야 합니다.
4. 초심자가 간단한 기능 수정, 필드 추가, 검증 추가, API 추가, 테스트 추가를 직접 할 수 있어야 합니다.
5. 언어 문법, 표준 라이브러리, 프레임워크, 외부 라이브러리, 빌드 도구, 프로젝트 자체 규칙을 명확히 구분해야 합니다.
6. 설명은 반드시 실제 프로젝트 코드와 연결해야 합니다.
7. 추상 설명보다 실제 파일 경로, 클래스명, 함수명, 코드 블록, 데이터 흐름을 우선해야 합니다.

# 2. 전체 작업 순서

문서를 생성하기 전에 반드시 다음 순서로 코드베이스를 분석하세요.

## 2-1. 파일 탐색 순서

다음 순서로 파일을 먼저 파악하세요.

1. 루트 디렉터리 구조
2. README 또는 기존 문서
3. 빌드 파일  
   예: `build.gradle`, `pom.xml`, `package.json`, `go.mod`, `Cargo.toml`
4. 실행 진입점  
   예: `main`, `Application`, `index`, `server`, `app`
5. 설정 파일  
   예: `application.yml`, `.env.example`, `Dockerfile`, `docker-compose.yml`
6. 라우팅/API/Controller/Handler 파일
7. Service/UseCase/Business Logic 파일
8. Repository/DAO/Database Access 파일
9. Entity/Model/DTO/Schema 파일
10. 공통 모듈  
    예: exception, config, util, middleware, filter, interceptor
11. 테스트 코드
12. 배포/CI 파일

## 2-2. 분석 우선순위

모든 파일을 똑같이 자세히 설명하지 마세요.  
다음 우선순위로 깊이를 조절하세요.

### 최상위 우선순위

- 실행 진입점
- 외부 요청 진입점
- 대표 기능 흐름에 직접 참여하는 파일
- 데이터 모델
- 설정 파일
- 에러 처리
- 테스트 코드

### 중간 우선순위

- 유틸리티
- 공통 상수
- 보조 설정
- 내부 헬퍼 함수
- 반복적으로 사용되는 공통 컴포넌트

### 낮은 우선순위

- 자동 생성 파일
- 빌드 산출물
- 단순 스타일 파일
- 사용되지 않는 코드
- 프로젝트 이해에 중요하지 않은 보조 파일

단, 낮은 우선순위 파일이라도 “왜 자세히 설명하지 않는지”는 짧게 언급하세요.

# 3. 문서 작성 공통 규칙

모든 문서에 다음 규칙을 적용하세요.

1. 실제 파일 경로를 반드시 명시하세요.
2. 실제 클래스명, 함수명, 메서드명, 변수명을 반드시 사용하세요.
3. 코드 블록은 큰 덩어리보다 의미 있는 작은 블록으로 나누세요.
4. 코드 설명은 “한 줄 단위” 또는 “키워드 단위”로 설명하세요.
5. 초심자가 모를 만한 키워드는 바로 설명하세요.
6. 언어 문법과 프레임워크 기능을 반드시 구분하세요.
7. 설명은 반드시 “개념 → 실제 코드 → 코드 해석 → 프로젝트에서의 역할 → 수정 포인트” 순서로 작성하세요.
8. “쉽다”, “간단하다”, “당연하다” 같은 표현은 사용하지 마세요.
9. 프로젝트에 없는 개념을 억지로 길게 설명하지 마세요.
10. 프로젝트 이해에 필요한 배경 개념은 충분히 설명하세요.
11. IDE를 켜지 않아도 구조가 머릿속에 그려지게 작성하세요.
12. 초심자가 직접 수정할 수 있는 지점을 반드시 포함하세요.
13. 수정 예시는 반드시 “수정 전 코드”와 “수정 후 코드”를 함께 보여주세요.
14. 코드가 너무 길면 전체를 붙이지 말고 핵심 블록만 인용한 뒤, 생략한 부분의 역할을 설명하세요.
15. 프로젝트 코드에 확실히 존재하지 않는 내용은 추측하지 말고 “코드에서 확인되지 않음”이라고 표시하세요.

# 5. 코드 설명 깊이 기준

코드는 다음 깊이로 설명하세요.

## 5-1. 핵심 코드

대표 기능 흐름에 직접 참여하는 코드는 매우 자세히 설명하세요.

설명 방식:

```text
코드 블록
→ 키워드별 설명
→ 줄 단위 설명
→ 이 코드가 호출되는 시점
→ 이 코드가 호출하는 다음 코드
→ 수정 가능한 지점
```

## 5-2. 보조 코드

유틸리티, 설정 보조, 단순 매핑 코드는 중간 깊이로 설명하세요.

설명 방식:

```text
역할
→ 입력
→ 출력
→ 어디서 호출되는지
→ 주의할 점
```

## 5-3. 중요도가 낮은 코드

프로젝트 이해에 큰 영향을 주지 않는 코드는 짧게 설명하세요.

설명 방식:

```text
역할
→ 자세히 보지 않아도 되는 이유
```

# 7. 출력 분량 제한과 분할 기준

한 문서가 너무 길어지면 안 됩니다.

다음 기준을 지키세요.

1. 하나의 Markdown 파일은 가능한 한 하나의 목적만 가져야 합니다.
2. `05-file-by-file.md`처럼 너무 길어질 문서는 반드시 디렉터리로 분리하세요.
3. 한 파일에서 여러 도메인을 다루면 도메인별 소제목을 명확히 나누세요.
4. 코드 블록은 필요한 핵심 부분만 포함하세요.
5. 전체 코드를 그대로 붙여넣지 말고, 초심자 이해에 필요한 핵심 블록을 선별하세요.
6. 단, 핵심 흐름을 이해하는 데 필요한 코드는 생략하지 마세요.
7. 생략한 경우에는 무엇을 생략했는지 설명하세요.

# 10. 최종 요청

이제 현재 프로젝트 코드베이스를 분석해서 위 구조에 맞는 문서 세트를 생성하세요.

반드시 다음 원칙을 지키세요.

- 한 파일에 모든 내용을 넣지 마세요.

- 실제 코드에 기반해 설명하세요.

- 문체는 친절하지만 얕지 않게 작성하세요.
